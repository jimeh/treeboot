Implementation plan

treeboot v1.9.0 command expansion

This plan turns the v1.9.0 spec additions into a focused implementation sequence for the CLI, public core crate, generated metadata, and tests. It is intentionally an implementation contract, not a replacement for docs/SPEC.md.

Spec: v1.9.0 Scope: five new commands Structured output: JSON and YAML Core API: command-shaped facades Generated assets: metadata and schema copy

Target Shape

treeboot status

Keep the existing discovery status behavior and add JSON and YAML output for editor, agent, and tooling integrations. Support --format json|yaml, --json, and --yaml.

Reporting output flags

Use --format text|json|yaml as the primary contract with --json and --yaml shortcuts on read-only reporting commands. schema keeps --output because its payload is already JSON.

treeboot version

Print human-readable version metadata by default and stable structured output with --format json|yaml, --json, or --yaml. It must not discover Git context, init scripts, or config.

treeboot check

Validate the selected bootstrap contract without executing scripts, applying file operations, or running configured commands. Print a success line by default and support JSON and YAML output from the beginning.

treeboot schema

Print the embedded config JSON Schema to stdout. The schema must be the same content published as the release config.schema.json, and --output / -o should write it directly to a file.

treeboot doctor

Print human troubleshooting diagnostics for Git availability, worktree discovery, root discovery, init script discovery, config discovery, and config validation. Support text, JSON, and YAML output in the initial implementation.

treeboot env

Print exactly the treeboot-owned child environment variables passed to init scripts and configured commands, sorted as KEY=value in text mode and as an object in structured output modes.

Public Core API

Keep command behavior in treeboot-core and presentation in the CLI crate. The CLI should mostly parse arguments, call these entrypoints, and render text, JSON, or YAML. Reports used by structured output commands should derive Serialize so the CLI does not invent separate programmatic shapes.

Command Core API Reason to expose
status inspect_status(StatusOptions) -> Result<StatusReport> Already exposed; derive Serialize for structured output.
version version_info(package, version) -> VersionInfo Callers should not duplicate the implemented spec version.
schema config_schema_json() -> &'static str The schema is generated from core-owned config semantics.
env inspect_env(EnvOptions) -> Result<EnvReport> Provides the exact child environment contract directly.
check check(CheckOptions) -> Result<CheckReport> Avoids reimplementing run selection and validation logic.
doctor diagnose(DoctorOptions) -> DoctorReport Programmatic consumers need all diagnostics, not only first error.

YAML dependency boundary

YAML serialization belongs in the treeboot CLI crate. treeboot-core should keep returning serializable report structs and avoid depending on CLI output-format libraries.

Generated Metadata

Embed metadata and the schema inside the core crate package boundary. Do not use build.rs to parse docs or generate schema at compile time.

Crate packaging constraint

Avoid include_str! paths that point outside crates/treeboot-core. They can work in a workspace build but fail once the core crate is packaged independently for crates.io.

Shared Run Selection

Extract the common selection and validation path before implementing check and doctor. This prevents the new commands from drifting away from run.

Implementation Sequence

Generated metadata foundation

  • Add core asset copy for config.schema.json.
  • Add generated metadata.rs.
  • Update mise run generate and check tasks.

Core read-only helpers

  • Expose VersionInfo and schema accessors.
  • Add EnvOptions and EnvReport.
  • Re-export new types from lib.rs.

CLI simple commands

  • Add version, schema, and env.
  • Add schema --output <path> and -o.
  • Add shared reporting output args for text, JSON, and YAML.
  • Add a CLI-only YAML serialization dependency after dependency review.
  • Keep formatting in the CLI crate.
  • Ensure shell completions include the new commands.

Status and env structured output

  • Derive Serialize for status and env reports.
  • Support status --format json|yaml, --json, and --yaml.
  • Support env --format json|yaml, --json, and --yaml.

Selection refactor

  • Extract shared run selection from run.rs.
  • Keep existing run output and side effects unchanged.
  • Preserve root-checkout and strict behavior.

Check command

  • Validate init-script precedence and config selection.
  • Fail on declarative run-validation errors.
  • Print treeboot: check ok on text success.
  • Support JSON and YAML output for successful reports.
  • Do not execute scripts, files, or commands.

Doctor command

  • Return a report with ordered diagnostics.
  • Support doctor --json and --yaml from the beginning.
  • Continue after recoverable warnings where possible.
  • Exit non-zero only when fatal diagnostics are present.

Testing Strategy

Area Coverage
Generated artifacts mise run generate:check verifies schema freshness, core asset freshness, and metadata freshness.
Status Tests for text, JSON, and YAML output; --json and --yaml shortcuts; and --format conflict handling.
Version CLI tests for text output, JSON and YAML output through --format and shortcuts, and unchanged --version behavior.
Schema CLI test that treeboot schema equals the embedded core schema and parses as JSON. Cover --output and -o, including replacement of an existing regular file and failure when the parent directory is missing.
Env Integration tests in a real linked worktree verifying sorted KEY=value output, JSON and YAML object output, and no config/script discovery.
Check Tests for valid config, invalid config, run-validation failure, init-script precedence, --no-init-script, and strict root-checkout behavior. Cover text success output plus JSON and YAML success output through both output flag styles.
Doctor Tests for healthy output, missing config warning, ignored non-executable init script, invalid config failure, and explicit config failure. Cover JSON and YAML output for healthy and fatal reports through both output flag styles.
Completions Existing completion tests should be extended to include all five new subcommands.

Validation Commands

For the implementation PR, use focused checks during development and the normal pre-handoff gate before final handoff. 

mise run generate:check
mise run test:core
mise run test:cli
mise run harness:check
mise run check

Use mise run verify only if the implementation touches release packaging, CI workflow behavior, or broader architecture docs.

Resolved Decisions

  • doctor supports JSON and YAML in v1.9.0 instead of waiting for a later diagnostic-model pass.
  • check prints a success line in text mode and supports JSON and YAML output for successful reports.
  • schema supports --output <path> and -o <path> from the initial implementation.
  • Structured-output reporting commands support --format text|json|yaml, --json, and --yaml. This applies to status, version, config, check, doctor, and env.
