Milestone 8 implementation plan

Manual File Operation Commands

Add treeboot copy, treeboot symlink, and treeboot sync as direct CLI entry points over the same normalized file operation model and file application engine already used by declarative config.

Observable contract

Definition Of Done

The spec already owns user-visible behavior. Implementation should only extend the spec if new edge cases are discovered.

Roadmap scope Spec contract Implementation consequence
Add manual subcommands copy, symlink, and sync run file operations without config discovery, init script discovery, or configured commands. Add CLI variants and route directly to run_file_operation, not through run.
Reuse normalized file operation model Manual commands normalize to the same shape as config file entries. The subcommand supplies operation. Build FileOperation values from CLI arguments, then share planning and execution code with declarative runs.
Require one or more sources Each manual command requires at least one source positional. Missing source files are optional unless --required is supplied. Let clap reject empty source lists. Preserve optional missing-source behavior in shared validation.
Support target normalization Without --target, target equals source. With one source, --target is exact. With many sources, --target is joined with each source value. Implement pure normalization helpers with unit coverage before wiring execution.
Expose operation flags --required applies to all three commands. --symlinks applies to copy and sync. --compare, --delete, and --no-delete apply to sync. Keep operation-specific clap args separate so invalid flag combinations fail as usage errors.
Root-path source completion Source completions list files and directories from the resolved root path, relative to that root. Use one shared candidate helper and shell-specific plumbing only where the generated scripts require it.

Starting point

Current Baseline

Milestones 5, 6, and 7 already left most internals needed for this work in place.

Already implemented

  • Copy, symlink, and sync execution.
  • Dry-run output for creates, updates, deletes, and skips.
  • Strict, force, compare, and sync delete behavior.
  • Config parsing into FileOperation.
  • Validation into PlannedFileOperation.

Main gap

The reusable file executor is crate-private and expects a RunPlan. Manual commands need a public core entry point that constructs a file-only plan without pretending a config file exists.

CLI state

crates/treeboot/src/main.rs currently has run, config, init, and completions. The root command flattens RunArgs, so adding manual subcommands must preserve treeboot as an alias for treeboot run. 

target flow:
  parse manual command args
  resolve RunContext from cwd and optional --root
  handle root-checkout strict behavior consistently with run
  normalize sources and target into FileOperation values
  validate duplicate targets, boundaries, strict sync, missing sources,
      and unsafe source symlinks
  apply file operations with strict, force, and dry-run options
  return a file-operation report

Shape of the change

Architecture

Keep treeboot as clap and reporting glue. Put manual operation normalization and execution in treeboot-core.

Core API

Add a public run_file_operation entry point, likely in a new manual.rs or file_operation.rs module exported from lib.rs. The API should take operation kind, root override, source list, target override, operation flags, and runtime flags. 

pub struct FileOperationOptions {
    pub cwd: Option<PathBuf>,
    pub root: Option<PathBuf>,
    pub operation: FileOperationKind,
    pub sources: Vec<PathBuf>,
    pub target: Option<PathBuf>,
    pub required: bool,
    pub symlinks: Option<SymlinkMode>,
    pub compare: Option<SyncCompare>,
    pub delete: Option<bool>,
    pub strict: bool,
    pub force: bool,
    pub dry_run: bool,
}

pub fn run_file_operation(
    options: FileOperationOptions,
    reporter: &mut dyn Reporter,
) -> Result<FileOperationReport>

Execution flow

  1. Parse manual CLI args in crates/treeboot.
  2. Resolve context with context::resolve.
  3. From the root checkout, report the same noop as run; under strict mode, fail like run.
  4. Normalize manual sources into FileOperations.
  5. Validate with the same boundary and duplicate checks.
  6. Apply with apply_file_operations.
  7. Return a small report for tests and library users.

DRY pass

Shared Implementation Rules

Manual commands are another input surface for the file operation engine, not a second implementation of file operation behavior.

Refactor before duplication

  • Extract common helpers when manual commands need config behavior.
  • Keep source/target normalization testable as a small unit.
  • Keep validation rules in one path shared by config and manual ops.
  • Keep file planning and application in the existing file engine.
  • Keep output events shared unless the spec requires new output.
  • Keep completion candidate generation in one shared helper.

Duplication to avoid

  • Do not reimplement duplicate target detection for manual commands.
  • Do not reimplement boundary checks in CLI code.
  • Do not reimplement strict, force, or dry-run file behavior.
  • Do not reimplement copy, symlink, sync, compare, or delete logic.
  • Do not fork completion candidate generation per command.
  • Do not create fake config files or TOML spans for CLI errors.

Preferred extraction

If plan_run_config is too config-shaped for manual commands, extract a shared file-only planner that accepts a RunContext, a slice of normalized FileOperations, and RunPlanOptions. Then make plan_run_config and run_file_operation call that same helper.

Source to target

Normalization Matrix

Manual operations must produce the same internal file operation shape as declarative config.

Input Source path Target path
treeboot copy a root/a worktree/a
treeboot copy a --target b root/a worktree/b
treeboot copy a c root/a, root/c worktree/a, worktree/c
treeboot copy a c --target b root/a, root/c worktree/b/a, worktree/b/c

Validation edge cases

  • Duplicate derived targets are fatal before side effects.
  • Absolute source paths must still pass source boundary checks.
  • Absolute target paths must stay inside the worktree.
  • --required turns missing source skips into errors.
  • --strict rejects all sync operations.

Flag defaults

  • copy required defaults to false.
  • sync compare defaults to metadata.
  • sync delete defaults to false.
  • copy and sync symlinks default to preserve.
  • force and dry-run default to false.

Shell UX

Completion Plan

Milestone 7 added script generation. Milestone 8 must make manual source completion resolve against the root path.

Static generation

  • Generated scripts must include copy, symlink, and sync.
  • Generated scripts must include all command-specific flags.
  • Unsupported shell behavior should remain a clap usage error.
  • Completion generation must remain side-effect free.

Dynamic candidates

  • Candidate root is resolved root path, including --root.
  • Candidate values are relative paths from the root path.
  • Directories should complete in a way the shell can continue.
  • Errors during candidate discovery should fail quietly.
  • Behavior must be implemented for every supported shell.
  • The root-relative candidate helper should be testable without a shell.

Implementation note

If the current clap_complete setup cannot express root-aware dynamic candidates uniformly across supported shells, add shell-specific generation or shared helper scripts as needed. The milestone is not complete until Bash, Zsh, Fish, PowerShell, and Elvish all support root-aware source file and directory completion.

Watch points

Risks And Constraints

Most implementation risk is in sharing validation correctly and in not overstating dynamic completion support.

  • Validation Manual commands must not bypass duplicate target, source boundary, target boundary, strict sync, or unsafe symlink checks.
  • Escape hatch Manual commands always apply boundary checks. Do not inherit dangerously_allow_targets_outside_worktree behavior from declarative config.
  • No discovery The manual path should resolve worktree/root context only. It must not discover config files, inspect init scripts, or execute configured commands.
  • Error origin Manual validation errors should identify CLI-originated file operations without misleading config paths, line numbers, or TOML column numbers.
  • Windows Symlink behavior differs by platform. Reuse existing file engine tests and gate symlink assertions the same way current tests do.
  • Completions Cross-shell dynamic completions may need shell-specific code, but all currently supported shells must get root-aware candidates.
  • DRY If manual behavior needs logic already used by config runs, refactor the shared implementation instead of duplicating that behavior in the manual command path.

Settled Decisions

No unresolved product or public API questions remain. Remaining choices are implementation details.

Completion requirement

Root-aware file and directory source completion is required for every shell already supported by treeboot completions. The implementation may use shared helpers, shell-specific generators, or both.

Public API naming

The public core entry point is run_file_operation. Naming for the options and report structs can follow the style that reads best beside RunOptions, RunReport, and InitOptions.
