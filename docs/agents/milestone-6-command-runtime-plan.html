treeboot roadmap

Milestone 6: command runtime

Replace the temporary declarative command gate with a real command runner. The target behavior is that validated config commands run after successful file operations, support shell and direct process forms, honor command-local environment and working directory, and batch consecutive async commands without weakening failure handling.

Status: planning document One PR recommended Spec first Core-owned behavior Std-first concurrency No new dependency expected

Milestone acceptance statement

A config containing only commands should execute. A config containing file operations and commands should complete file operations first, then execute commands. If file operations fail, no configured command should start. If commands are skipped or dry-run, file operation behavior should be unchanged.

Starting point

Current Baseline

Most of the declarative command model already exists. Milestone 6 should consume it rather than reshaping parsing.

Already implemented

  • commands and [[command]] parsing.
  • Shell vs direct invocation normalization.
  • cwd, env, async, and allow_failure fields.
  • Command cwd boundary validation.
  • Treeboot-owned env override rejection.

Temporary gate

run.rs currently returns CommandExecutionNotImplemented before file mutation whenever commands exist and --skip-commands is not set. This guard is the main removal point.

Useful existing shape

RunPlan.commands already carries the validated runtime-ready model. The new executor should take a RunPlan, not raw config entries. 

Current run flow:
  load config
  build RunPlan
  if commands exist and !--skip-commands:
      fail before file mutation
  apply file operations

Milestone 6 run flow:
  load config
  build RunPlan
  apply file operations
  if !--skip-commands:
      run planned commands or report dry-run commands

Recommended choices

Implementation Decisions

These are the proposed decisions to review before writing the runtime. Changing any of them should update the spec first.

Decision Recommendation Reason
PR shape One PR, internally staged as docs, sequential runtime, then async batching. The milestone is one behavioral contract. Separate merged PRs would create awkward intermediate states.
Runtime owner Add crates/treeboot-core/src/commands.rs. The CLI should stay a thin adapter. Process execution is reusable core behavior, like file operations.
Concurrency Use the standard library. No async runtime dependency. The required model is child-process wait-all batching plus line-buffered output for async commands. A bounded set of reader threads and channels is enough and keeps distribution small.
Child output Sequential commands inherit stdout and stderr. Async commands pipe stdout and stderr, line-buffer complete lines, and serialize prefixed output. Sequential tools keep natural terminal behavior. Async output avoids mid-line interleaving between parallel commands.
Failure reporting Record all fatal failures in an async batch, then return a typed error listing every failed command label after every child has exited. This matches the wait-all contract and avoids orphaned or prematurely abandoned async commands.
Allowed failures Emit a warning-style event and keep going. Non-zero allowed commands are still useful operational signals; silent success would hide important setup state.
Dry-run Report each command that would run, including async grouping, and spawn nothing. Dry-run should preview command runtime decisions without changing the worktree or invoking external programs.

Code shape

Architecture

The executor should sit after file operations in run.rs and consume the existing validated plan.

Core API sketch

pub(crate) struct CommandRunOptions {
    pub(crate) dry_run: bool,
}

pub(crate) fn run_commands(
    plan: &RunPlan,
    options: CommandRunOptions,
    reporter: &mut dyn Reporter,
) -> Result<()>

Return shape

The runner can return Result<()> initially. A richer report is only worth adding if tests or output need command counts, skipped counts, or duration later.

Execution model

Runtime Algorithm

Keep the top-level flow serial and easy to reason about. Only consecutive async commands introduce concurrency.

Run flow
1. Validate Load config and build RunPlan as today.
2. Files first Apply or dry-run file operations. Stop on failure.
3. Command gate If --skip-commands, return success after files.
4. Runtime Execute or dry-run planned commands in declaration order.
Command flow
1. Partition Treat non-async commands as singletons. Group consecutive async commands into batches.
2. Build process Choose shell or direct invocation. Apply cwd and merged env.
3. Execute Spawn and wait. For async batches, start every child before waiting for failures.
4. Classify Allowed failures warn and continue. Fatal failures stop after the singleton or after all batch children finish. 
run_commands(plan, options, reporter):
  for item in partition_commands(plan.commands):
    if options.dry_run:
      report would-run events
      continue

    if item is sequential:
      status = run_one(command)
      handle_status(command, status)

    if item is async batch:
      children = spawn every command in batch
      results = wait for every child
      fatal = results where status failed and !allow_failure
      if fatal is not empty:
        return Err(CommandBatchFailed(fatal))

No fail-fast inside async batches

The spec says treeboot waits for every command in an async batch before exiting non-zero. The implementation should not kill or skip sibling commands just because one command fails quickly.

Command labels

Show both name and invocation when name is present: Install packages: npm install. Without a name, format shell commands as the shell string and direct commands as program arg....

Child stdio

Sequential commands inherit stdout and stderr. Async batch commands pipe both streams, line-buffer complete lines, prefix each line with the command label, and serialize writes so command output does not interleave mid-line.

Spec update required

docs/SPEC.html should explicitly state the lifecycle output, sequential inherited stdio, async line-buffered output, allowed-failure warning, and async failure aggregation behavior before the runtime lands.

Validation

Testing Strategy

Tests should cover core command semantics and CLI-visible behavior from real linked worktrees.

Core tests

  • Build shell commands with platform-specific shell.
  • Build direct commands without shell parsing.
  • Merge treeboot env and per-command env.
  • Use planned cwd for process execution.
  • Dry-run emits would-run events and spawns nothing.
  • Allowed non-zero exits warn and continue.
  • Fatal singleton failure stops later commands.
  • Consecutive async commands run as one batch.
  • Async batch waits for all children before returning failure.
  • Async output line buffering never emits partial mixed lines.
  • Async trailing output without a newline is flushed at EOF.
  • Async stdout lines stay on stdout and stderr lines stay on stderr.
  • Async batch failures list every fatal command label.

CLI integration tests

  • Command-only config executes from a linked worktree.
  • File operations run before commands.
  • Failed file operation prevents command execution.
  • --skip-commands still applies files only.
  • --dry-run reports files and commands without mutation or process side effects.
  • Command env can observe TREEBOOT_ROOT_PATH, TREEBOOT_WORKTREE_PATH, and local env values.
  • Direct program with args preserves argument boundaries.
  • Async commands complete before a following sequential command starts.
  • Fatal command failure returns the runtime failure exit code.
  • Command failure after successful file operations does not roll back files.

Test harness notes

Prefer small scripts or current-test-executable helper modes over external tools like npm or mise. Tests should write marker files in the temp worktree to prove ordering, cwd, env, and dry-run behavior.

Avoid timing-sensitive assertions where possible. Use files, explicit blocking points, and ordered marker writes to prove async grouping and wait-all behavior.

Watch points

Risks And Mitigations

These are the main ways command runtime can become flaky or surprising.

Output noise Inherited output is best for sequential tools, but parallel inherited output can interleave mid-line. Capture only async command output, line-buffer it, and prefix each complete line.
Async leaks Returning before every async child exits would violate the spec. Keep child handles in a batch structure and wait all before classifying fatal failures.
TTY behavior Async commands with piped output may disable color or progress UI because stdout and stderr are not TTYs. Accept that tradeoff for deterministic combined logs; sequential commands keep inherited terminal behavior.
Platform shell Unix and Windows shell invocation differs. Keep the shell builder behind platform cfgs and use direct-command tests for portable argument-boundary assertions.
Test flakiness Timing-based async tests can be brittle. Prefer marker files and ordered dependencies over short sleeps where possible.
Partial side effects File operations intentionally happen before commands. Tests should make clear that command failure does not roll back file changes.
Label ambiguity Friendly command names help humans, but errors still need enough invocation detail to debug the failing command. Format both carefully in the output contract.

Review items

Resolved Questions

These decisions came out of plan review and should be reflected in the spec before implementation.

  1. Sequential commands inherit stdout and stderr. Async commands use line-buffered capture with serialized, prefixed output.
  2. Allowed failures always produce a warning line.
  3. Async batch failure errors list every failed command label.
  4. Command output includes both name and invocation when a name is present.
  5. A start event plus child output is enough for normal successful runs. Do not add a success event in milestone 6.

Settled recommendation

Use inherited stdio for sequential commands and std-threaded line-buffered output for async batches. Warn on allowed failures, list every failed async command label, include both name and invocation, and skip success events.
