File Operation Progress Output Plan

Plan for concise default file-operation output, verbose detailed output, and interactive progress rendering for large copy and sync trees.

Decision

Use indicatif in the treeboot CLI crate for interactive progress. Keep treeboot-core terminal-agnostic by emitting structured lifecycle, progress, and summary events. Default output should be one summary line per top-level file operation. A new verbose mode should preserve today's per-file/per-action output.

Problem

The current file-operation engine expands directory copy and sync work into concrete actions, then reports each concrete action. That is useful for small trees but too noisy for large directories such as node_modules. The default experience should tell users what happened without flooding terminal logs.

Large trees also have two visible phases. Planning can spend meaningful time walking the source and target trees before any mutation starts. Execution then applies the concrete actions discovered during planning. Users should see progress in both phases when running interactively.

Goals

  • Default to concise output: one final summary line per top-level file operation that does work or skips.
  • Show an interactive spinner while planning copy and sync trees.
  • Show an interactive progress bar while executing planned actions.
  • Keep non-interactive logs clean: no control sequences, spinner frames, or partial lines in CI or redirected output.
  • Preserve detailed output behind -v / --verbose.
  • Keep warnings visible by default, even when detailed action output is hidden.
  • Keep treeboot-core usable by embedders without terminal UI dependencies.

Non-goals

  • Do not introduce a full-screen TUI or a persistent dashboard.
  • Do not manually render carriage-return progress lines.
  • Do not make command child output go through progress rendering.
  • Do not hide destructive sync deletes in summary mode.
  • Do not add progress output to JSON or YAML inspection commands.

User-facing Contract

Default text output

Default output should emit one final line for each top-level file operation that had a visible decision. Examples: 

treeboot: copy .env -> .env
treeboot: copy node_modules -> node_modules (1842 changed, 27 skipped)
treeboot: sync shared -> shared (4 changed, 1 deleted)
treeboot: skip copy .env.local; missing source

Unchanged sync operations should stay silent, matching the existing contract for unchanged sync files and directories.

Manual copy and sync commands should follow the same default output contract as declarative config execution. A manual command with multiple source arguments normalizes into multiple top-level file operations, so each source should get its own progress lifecycle and final summary: 

treeboot copy a b --target local

treeboot: copy a -> local/a (12 changed)
treeboot: copy b -> local/b (4 changed, 2 skipped)

Do not aggregate a multi-source manual command into one combined progress bar or one combined summary by default. Per-source summaries match the normalized operation model and make failures easier to locate.

Dry-run output

Dry-run should use the same summary shape, but with would wording: 

treeboot: would copy .env -> .env
treeboot: would copy node_modules -> node_modules (1842 changes, 27 skips)
treeboot: would sync shared -> shared (4 changes, 1 delete)

Verbose output

-v / --verbose should print detailed events for every concrete action, preserving the current behavior as closely as possible: 

treeboot: copy node_modules/.bin/vite -> node_modules/.bin/vite
treeboot: copy node_modules/vite/package.json -> node_modules/vite/package.json
treeboot: skip copy node_modules/react; target exists

Verbose mode should not also print grouped summaries unless the implementation needs a final operation footer. Avoid duplicate noise by default.

Interactive progress

For TTY runs in non-verbose mode, the CLI should render ephemeral progress on stderr and keep final summaries on stdout. The rough terminal experience should be: 

treeboot: copy node_modules -> node_modules planning ...
treeboot: copy node_modules -> node_modules [##########-----] 1234/1842
treeboot: copy node_modules -> node_modules (1842 changed, 27 skipped)

The spinner and progress bar should be cleared or replaced cleanly before the final summary line is printed. Non-TTY output should include only the final summary lines.

For single concrete actions, such as copying one file to one file, progress UI will usually be too brief to matter. The final line should remain the concise current-style output without a parenthesized count because the source and target already contain all useful information.

CLI Surface

Command New option Behavior
treeboot / treeboot run -v, --verbose Detailed file-operation output. Command output remains inherited as today.
treeboot copy -v, --verbose Detailed output for manual copy actions. Default output and progress are summarized per normalized source operation.
treeboot symlink -v, --verbose Detailed output for manual symlink actions. Progress UI will rarely matter because symlink operations are single-action.
treeboot sync -v, --verbose Detailed output for manual sync actions. Default output and progress are summarized per normalized source operation.

Do not add a user-visible --progress mode in the first pass. Use automatic TTY detection through indicatif. A future --progress auto|always|never can be added if users need to force or suppress progress explicitly.

Dependency Plan

Add indicatif to the treeboot CLI crate, not treeboot-core. This follows the dependency boundary in docs/agents/dependencies.md: terminal presentation belongs in the CLI crate. indicatif handles terminal detection, hidden progress for non-interactive targets, refresh throttling, stderr drawing, and clean finish behavior.

Before finalizing the dependency, inspect its current transitive dependencies and MSRV impact: 

cargo tree -p treeboot
cargo tree -p treeboot-core
mise run check

Keep treeboot-core free of indicatif, console, ratatui, and terminal color/styling dependencies.

Core Event Model

Extend OutputEvent with structured events that describe planning, execution, and summaries without requiring terminal rendering in core. 

pub enum OutputEvent {
    FileOperationPlanningStarted {
        operation: FileOperationKind,
        source: PathBuf,
        target: PathBuf,
    },
    FileOperationPlanningFinished {
        operation: FileOperationKind,
        source: PathBuf,
        target: PathBuf,
        action_count: usize,
    },
    FileOperationExecutionStarted {
        operation: FileOperationKind,
        source: PathBuf,
        target: PathBuf,
        action_count: usize,
    },
    FileOperationActionAdvanced {
        operation: FileOperationKind,
        source: PathBuf,
        target: PathBuf,
        action: FileOperationProgressAction,
    },
    FileOperationFinished {
        operation: FileOperationKind,
        source: PathBuf,
        target: PathBuf,
        summary: FileOperationSummary,
        dry_run: bool,
    },
    ...
}

The exact type names can change during implementation. The important contract is that core reports enough structured data for the CLI to render either a spinner/bar or plain text without parsing formatted strings.

Summary shape

Add a compact summary type in core: 

pub struct FileOperationSummary {
    pub changed: usize,
    pub skipped: usize,
    pub deleted: usize,
    pub warnings: usize,
    pub unchanged: usize,
}

Suggested counting rules:

  • changed: create directory, copy file, create symlink, or replace file/symlink actions.
  • skipped: optional missing sources and existing targets skipped by default copy/symlink behavior.
  • deleted: sync target-only delete actions.
  • warnings: emitted warning events.
  • unchanged: optional for future use. The current planner does not create concrete actions for unchanged sync entries, so this can remain zero or omitted in the first pass.

Execution Model

Today apply_file_operations plans all actions for all operations, then applies the flat action list. To support per-operation progress and summaries, make planning and application preserve the top-level file operation boundary. Declarative config entries and normalized manual source arguments should use the same grouped execution model.

Step 1: Group planned actions by operation

Replace the single flat action list with a grouped plan: 

struct PlannedFileOperationActions {
    operation: FileOperationKind,
    source: PathBuf,
    target: PathBuf,
    actions: Vec<FileAction>,
}

Planning should report FileOperationPlanningStarted before walking a top-level copy/sync tree. After walking, report FileOperationPlanningFinished with the concrete action count. Symlink operations can still use the same lifecycle events for consistency, but the CLI can choose not to show a spinner for fast single-action work. Manual multi-source commands should produce one PlannedFileOperationActions group per normalized source, not a command-wide aggregate group.

Step 2: Apply one group at a time

For each grouped operation:

  1. Report execution start with total action count.
  2. Apply or dry-run each action.
  3. In verbose mode, report the detailed existing FileApplied, FileWouldApply, FileSkipped, FileWouldSkip, FileDeleted, and FileWouldDelete events.
  4. In summary mode, suppress detailed action events except warnings.
  5. After each concrete action, report an advance event so the CLI progress bar can increment.
  6. Report the final grouped summary.

If an operation fails midway, ensure the reporter clears active progress before the error is printed by the CLI. Prefer RAII/drop cleanup in the CLI progress state so unexpected errors do not leave partial terminal lines.

Step 3: Preserve safety behavior

Do not weaken existing validation or apply-time checks. The refactor must keep:

  • validation before side effects for duplicate/overlapping targets, target boundaries, required missing sources, strict sync, and unsafe source symlinks;
  • apply-time ancestor symlink checks before mutating each target;
  • sync delete behavior only when explicitly enabled;
  • dry-run non-mutation guarantees.

Reporter Design

Replace StdoutReporter with a CLI reporter that can manage progress state while still implementing the same Reporter trait. 

struct CliReporter {
    progress: Option<ProgressState>,
    verbose: bool,
}

enum ProgressState {
    Planning(ProgressBar),
    Executing(ProgressBar),
}

The CLI reporter should:

  • create an indicatif spinner on planning-start events when progress is enabled;
  • finish and clear the spinner on planning-finished events;
  • create an indicatif progress bar on execution-start events when action count is greater than one;
  • increment the bar on action-advanced events;
  • finish and clear the bar before printing the final summary line;
  • print warnings through ProgressBar::suspend or an equivalent cleanup path so warnings do not corrupt the progress line;
  • print normal final output to stdout, preserving current treeboot: line prefixes.

Progress enablement

Let indicatif handle most terminal detection. Use its normal stderr draw target. Do not force progress when stdout/stderr are redirected. Keep progress disabled in verbose mode to avoid interleaving detailed lines with progress bars.

Formatting Rules

Summary wording should be deterministic and easy to assert in tests.

Summary Output
Single changed action treeboot: copy .env -> .env
Single dry-run action treeboot: would copy .env -> .env
Multiple changed actions treeboot: copy shared -> shared (12 changed)
Changed plus skipped treeboot: copy shared -> shared (12 changed, 3 skipped)
Sync delete treeboot: sync shared -> shared (4 changed, 1 deleted)
Only skip treeboot: skip copy .env.local; missing source
Dry-run summary treeboot: would copy shared -> shared (12 changes)

Omit parenthesized counts for exactly one visible decision because the operation line already carries the useful source and target information. Use parenthesized counts only when they summarize grouped or expanded work. Use singular nouns for count one and plural nouns otherwise. Keep skip wording for single skipped top-level operations so existing small-file output remains familiar.

Testing Strategy

Core unit tests

  • Directory copy default mode emits one summary event for many child actions.
  • Directory copy verbose mode emits detailed child action events.
  • Directory copy dry-run emits a would copy summary and does not mutate files.
  • Sync delete summary includes delete counts.
  • Unchanged sync remains silent in summary mode.
  • Warnings still emit in summary mode.
  • Failures during apply clear progress lifecycle by event ordering and still return the original typed error.

CLI integration tests

  • treeboot run defaults to grouped output for directory copy/sync.
  • treeboot run --verbose prints concrete child actions.
  • treeboot copy --verbose and treeboot sync --verbose pass verbose mode into core.
  • Manual multi-source copy and sync commands emit one compact summary per normalized source operation, not one command-wide aggregate summary.
  • Single-file copy and sync summaries omit parenthesized counts.
  • Non-TTY command assertions contain no spinner frames, progress bars, or ANSI/control output.
  • Dry-run output uses summary wording and avoids mutation.

Progress rendering tests

Avoid brittle visual assertions against indicatif templates. Prefer tests around reporter state transitions with hidden draw targets or injected progress mode. If a full terminal-rendering smoke test is needed, keep it narrow and do not require exact frame timing.

Risks And Mitigations

Risk Mitigation
Progress output pollutes CI logs. Use indicatif hidden behavior for non-terminals and keep final summaries as plain stdout lines.
Command child output corrupts progress bars. Progress only covers file operations. Clear or finish bars before command execution starts.
Warnings get hidden in summary mode. Always emit warnings. Suspend or clear progress before printing warning lines.
Grouped summaries hide useful debugging detail. Preserve current detailed output under --verbose.
Planning still feels slow for huge trees. Emit planning lifecycle events before recursive walks and render a spinner immediately in interactive mode.
Refactor changes mutation ordering. Keep existing child traversal order inside each top-level operation and add regression tests around sync delete, force, and dry-run behavior.

Open Questions

  • Should --verbose also keep progress disabled explicitly, or should progress be enabled while detailed events are printed above the bar? Recommendation: disable progress in verbose mode.
  • Should zero-action top-level copy operations emit an explicit unchanged/empty summary? Recommendation: preserve current behavior where unchanged sync is silent and existing-target copy skips still report.
  • Should progress be forceable later with --progress always|never? Recommendation: defer until a real caller needs it.
