treeboot roadmap

Milestone 5: declarative file operations

Implement the file-effect portion of declarative config execution while preserving the existing validation-first contract. The target outcome is that file-only configs can apply, dry-run, and report planned work; command runtime remains explicitly gated for milestone 6.

Status: ready for implementation Spec v1.3.0 baseline Spec is the contract Core-owned behavior Std-first implementation

Contract map

Spec Alignment

Roadmap milestone 5 is short. The implementation should use these spec clauses as the detailed acceptance criteria.

Roadmap scope Spec contract Implementation consequence Status
Copy files and directories Directory copies recurse into the configured target path and never remove target-only files. Build recursive copy actions. Force may overwrite matching files inside an existing target directory, but copy must not become sync. Build
Create relative symlinks when possible Symlink operations create links from the target path to the source, using a relative link when it can be computed. Add a small relative-path helper in core. Fall back to an absolute source path only when relative computation is not possible. Build
Sync files/directories Sync reconciles targets, compares by metadata or checksum, and preserves target-only files by default for directories. Build sync as a separate traversal. Checksum mode should be a streaming byte comparison unless a later need justifies a digest dependency. Metadata mode should preserve source mtime after copying to keep future runs idempotent. Deleting target-only files must require explicit delete = true or --delete. Build
Default, strict, force, dry-run conflicts Default skips existing copy/symlink targets. Strict fails copy/symlink conflicts and rejects sync. Force replaces files and symlinks but must not erase directories casually. Decide conflict outcomes before mutation. Dry-run reports the same decisions without applying them. Build
Preserve safe source symlinks Copy and sync preserve safe source symlinks. A symlink is unsafe if empty or resolving outside TREEBOOT_ROOT_PATH. Keep the project root as the safety boundary. Rewrite copied symlinks only when needed to keep them pointing at the analogous worktree destination, and warn when the final target does not exist or will not be created. Build

Symlink rewrite checklist

The compatibility behavior lives in docs/SPEC.html. Implement preserved source symlinks with this sequence.

  1. Read the raw source symlink target without following it.
  2. Resolve that target from the source link parent.
  3. Reject it if the resolved target is outside the root path.
  4. Map root-relative resolved targets to the analogous worktree path.
  5. Write a relative link from the destination parent when possible.
  6. Preserve the raw link text when no rewrite is needed, and warn if the final target does not exist or will not be created.

Scope control

Milestone Boundary

The clean boundary is file effects only. Command execution should remain unavailable until milestone 6.

Current implementation baseline

The spec and roadmap contract changes have already landed. Current code parses and normalizes file operations, supports delete, accepts --skip-commands, and validates duplicate targets, boundaries, required sources, strict sync rejection, and unsafe source symlinks before side effects. Milestone 5 starts from that validation layer.

Runtime execution still stops at ConfigExecutionNotImplemented. The milestone should add the file executor, file output events, and the more specific CommandExecutionNotImplemented gate for configs that still contain commands.

File-only config

Validate the full config, apply file operations, report results, and exit 0 when all file effects succeed.

Config with commands

If --skip-commands is not set, fail before file side effects with a new CommandExecutionNotImplemented error. This avoids mutating a worktree and then failing only because milestone 6 has not landed.

Dry run

Build and report the same planned file decisions, but write nothing. Validation errors still exit non-zero and must not produce partial output that implies mutation happened.

Manual commands remain out of scope

Milestone 5 implements declarative file-operation execution only. treeboot copy, treeboot symlink, and treeboot sync are milestone 8, so the reusable engine can stay crate-private until that public command surface is ready.

Execution model

Execution Algorithm

Prefer an explicit action plan. It makes dry-run honest and keeps validation failures ahead of mutation.

Run flow
1. Discover Load init script or config exactly as today.
2. Validate Parse config and build RunPlan.
3. Gate commands If commands exist and not --skip-commands, fail before file effects.
4. Apply files Plan actions, emit dry-run or apply output, return success.
File action flow
1. Inspect source Use symlink_metadata so symlinks stay visible.
2. Inspect target Missing targets become creates; existing targets become conflict decisions.
3. Build actions Expand directory copy and sync into per-entry actions so dry-run reports exact creates, updates, deletes, warnings, and skips.
4. Apply serially Preserve operation order and stop on the first runtime failure. 
run()
  resolve context
  discover script/config
  load config
  plan_run_config()
  if plan.commands is not empty and !options.skip_commands:
      return Err(CommandExecutionNotImplemented)
  apply_file_operations(plan.files, FileApplyOptions)
  return RunReport { action: ConfigApplied }

Output Contract

Output should be structured in core and formatted in one place, matching the examples in the spec.

Proposed events

OutputEvent::FileApplied { operation, source, target }
OutputEvent::FileSkipped { operation, target, reason }
OutputEvent::FileWouldApply { operation, source, target }
OutputEvent::FileWouldSkip { operation, target, reason }
OutputEvent::FileDeleted { path }
OutputEvent::FileWouldDelete { path }
OutputEvent::FileWarning { path, reason }

Example lines

treeboot: copy .env -> .env
treeboot: skip .env.local; missing source
treeboot: skip .tool-versions; target exists
treeboot: would sync shared/config -> .config
treeboot: delete .config/old.toml
treeboot: warning: symlink .config/theme target does not exist

Keep events semantic

Avoid output variants that expose implementation details such as CopyFileAction. Events should describe what the user cares about: operation, source, target, and reason.

Delete events are only emitted for sync operations when deletion is explicitly enabled with delete = true or --delete.

Work order

Rollout Sequence

This order starts from the landed contract and existing validation, then adds dry-run safety before enabling mutations.

Phase 1: baseline and preflight

  1. Confirm the current spec and roadmap are the baseline.
  2. Reuse plan_run_config as the validation layer.
  3. Add command-runtime gate before file side effects.
  4. Add file action data types and dry-run action planning.

Phase 2: mutation primitives

  1. Implement parent directory creation.
  2. Implement copy file, copy directory, and preserved symlink.
  3. Rewrite copied symlinks when target layout changes.
  4. Implement symlink operation with relative target.
  5. Implement replacement rules for force mode.

Phase 3: sync

  1. Implement file sync compare modes.
  2. Preserve mtime after copying regular files.
  3. Implement directory reconcile traversal.
  4. Implement explicit target-only deletion and no-delete default behavior.

Phase 4: integration polish

  1. Wire output events into CLI stdout.
  2. Replace not-implemented run tests with file-runtime tests.
  3. Run mise run check.
  4. Document any discovered edge cases in agent docs.

Risk register

Risks

These are the places most likely to create behavior drift from the spec.

Partial mutation Runtime I/O can still fail after preflight. Keep validation failures ahead of mutation, but do not pretend file application is transactional. Stop on first failure and report the path.
Symlink portability Windows symlink creation may require privileges. Gate tests with platform cfg where needed and keep cross-platform behavior in the spec visible.
Metadata compare If copied files do not preserve modified time, metadata sync will update forever. Use std::fs::FileTimes through an opened file after copying regular files, and preserve regular file permissions where the platform supports it.
Directory force Force mode is destructive, but the spec explicitly says it must not erase directories casually. Directory type conflicts should fail instead of trying to be clever.
Race after preflight Filesystem state can change between planning and apply. Re-check target type before destructive replacements and return typed, path-specific runtime errors.
Checksum semantics Rust std has no stable content digest helper. Treat checksum mode as streaming byte equality for milestone 5 unless a real digest requirement appears.

Decisions

Resolved Decisions

These decisions close the planning questions and should be treated as implementation constraints.

Command gate wording

Use the more accurate CommandExecutionNotImplemented error for configs that contain commands when --skip-commands is not set. Keep the failure before file side effects.

Sync unchanged output

Stay silent for unchanged sync files and unchanged sync directories. Report creates, updates, and explicit deletes because those are the meaningful effects.

Sync delete default

Preserve target-only files by default in declarative and manual sync. Use delete = true in config or --delete / -D for manual sync to opt into deleting target-only files.

Copy existing directory default

In default mode, recursively copy all missing files from the source directory and skip anything that already exists. Treat this like manually listing every file in the directory rather than one single target conflict.

Public API surface

Keep milestone 5 internals crate-private. Public API shape can wait until milestone 8, when manual file-operation commands need to reuse the engine and the unfinished product is closer to a releasable contract.

Metadata preservation

Preserve regular file contents, permissions, and modified time during copy and sync. Metadata mode compares size plus modified time, so mtime preservation is required for idempotent reruns.

Checksum comparison

Implement checksum comparison as streaming byte-for-byte equality rather than adding a digest dependency. The result is the same update decision without expanding the dependency surface.

CLI shorthand flags

Use short aliases for frequent options: -r/--root, -c/--config <path>, -o/--format, -J/--json, -S/--strict, -f/--force, -n/--dry-run, -t/--target, -D/--delete, -s/--script, and -p/--path. Keep --skip-commands, --required, --symlinks, --compare, --no-delete, and init --config long-only.
